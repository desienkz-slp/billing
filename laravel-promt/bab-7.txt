# Bab 7: Keamanan, Konfigurasi, & Deployment

Sebagai sistem yang berurusan langsung dengan tagihan finansial (Billing) dan konektivitas perangkat pelanggan (MikroTik), LadaPala-Bill wajib mematuhi standar keamanan aplikasi web modern. Pada bab terakhir ini, kita akan membahas tembok pertahanan (Security Architecture), cara sistem membaca konfigurasi lingkungan, serta metode *Deployment*.

---

## 7.1 Lapis Keamanan dan Autentikasi (Security Layers)

Setiap *endpoint* atau halaman yang menampilkan antarmuka wajib mengikutsertakan berkas `inc/auth_guard.php`. Berkas ini adalah garda terdepan sistem keamanan.

### 7.1.1 Manajemen Sesi (Session Handling)
Saat Sesi PHP diinisialisasi, sistem secara otomatis memasang atribut parameter _cookie_ yang ketat:
*   `httponly = true`: Mencegah token sesi dicuri oleh serangan XSS (*Cross-Site Scripting*) via Javascript (`document.cookie`).
*   `samesite = 'Lax'`: Melindungi aplikasi dari serangan CSRF dengan membatasi skop pengiriman *cookie* antarsitus.
*   `secure`: Secara otomatis menjadi `true` jika aplikasi mendeteksi konfigurasi *proxy* HTTPS/TLS aktif.

### 7.1.2 Pemblokiran Pembajakan Permintaan (CSRF & Rate Limiting)
*   **CSRF Protection**: Seluruh *endpoint* REST API (seperti pada saat penekanan tombol bayar di `billing_payment_save.php`) mewajibkan adanya `csrf_token` di dalam badan JSON (*payload*). Jika token tidak cocok dengan yang dicetak di awal sesi, *backend* mengembalikan *HTTP 403 Forbidden*.
*   **Rate Limiting**: Berkas `inc/rate_limit.php` mencegah serangan *Brute Force* atau klik-berulang ganda (Double Submit). *Endpoint* dibatasi per menit berdasarkan alamat IP pengguna.

### 7.1.3 RBAC Dinamis (Role-Based Access Control)
Berbeda dengan sistem lama yang memeriksa hak akses menggunakan klausa `if ($role == 'admin')`, LadaPala-Bill menggunakan _Capability-based Access_. 
Fungsi seperti `auth_require_permission('can_delete')` akan menerjemahkan nama kapabilitas, memanggil fungsi penolong `perms_can_delete()`, lalu memverifikasi nilai *boolean* di dalam basis data untuk pengguna (User) tersebut. 
Pendekatan ini sangat solid; bahkan jika _hacker_ memalsukan perambannya untuk mengirim paket POST "Delete", sistem akan menolak pada baris awal.

## 7.2 Konfigurasi Sistem dan Enkripsi Kredensial

Aplikasi menghindari *hard-coding* kata sandi di dalam sumber kodenya (Source Code). 
Sistem memusatkan pemanggilan pengaturan ke **`inc/config.php`**.

1.  **Variabel Lingkungan (Environment Variables)**: Konfigurasi utama di-muat dari fail `.env`. Parameter pangkalan data seperti `BILLING_DB_HOST` dan sandinya dibaca pada saat *Runtime*.
2.  **Enkripsi Dekripsi Simetris (AES-256-CBC)**: Fitur yang sangat tidak lazim namun superior. Jika administrator merasa peladen *Cloud* miliknya rentan, ia bisa memasukkan kata sandi ke dalam `.env` dengan format awalan `ENC:`.
    Fungsi penolong `_crypt_env()` akan mendekripsi kata sandi tersebut menggunakan algoritma `aes-256-cbc` dipadukan dengan *Passphrase* *sha256* sebelum koneksi basis data dilakukan.
3.  **God Mode & Lisensi (Kill Switch)**: `config.php` mengekspos konstanta bawaan (Built-in) seperti `MASA_AKTIF` dan `MAX_PELANGGAN`. Jika peranti lunak (Software) telah melewati masa berlaku (*expired*), maka parameter _Boolean_ `GOD_MODE` akan dipicu ke _true_, yang akan merombak beberapa otorisasi *bypass* (kemungkinan berkaitan dengan *API* pusat atau _Remote Assistance_ vendor pengembang).

## 7.3 Deployment & Setup (Penyebaran Aplikasi)

Untuk mengadopsi LadaPala-Bill pada ISP baru, pengguna tidak perlu meng-impor fail `.sql` secara manual melalui *phpMyAdmin*. Sistem memfasilitasi _Wizard_ Instalasi (Deployment).

1.  **Fail Instalasi (`setup.php` & `setup-api.php`)**: Jika aplikasi dijalankan dan basis data kosong (atau `.env` belum ada), `inc/billing_db.php` akan mencegat (*intercept*) *traffic* pengunjung ke halaman Setup. Halaman GUI ini akan memandu pembuatan basis data dan menyuntikkan (inject) pengguna super-admin (root) pertama kalinya.
2.  **Manajemen Pencadangan (Backup Batch)**: Di direktori *root*, terdapat berbagai *script* untuk sistem operasi seperti `backup-TOTAL.bat` dan `backup_cloud.bat`. Administrator ISP cukup mengklik *batch* ini pada peladen Windows (*Local Node*), dan ia akan membungkus basis data MySQL berserta arsip sistem menjadi format kompresi (contoh: `backup_CLOUD_20260529_053923.zip`), menjaga keberlangsungan (Disaster Recovery).

## Kesimpulan Eksekutif (Executive Summary)

Sistem Informasi Penagihan Pembayaran Berbasis Web (SISPEMB / LadaPala-Bill) terbukti melampaui standar aplikasi manajemen ISP menengah. 
Melalui perancangan 7 Bab yang telah didokumentasikan ini, kita melihat perpaduan:
- Arsitektur **Front-Controller** (Bab 1)
- Skema **Hybrid RDBMS MySQL** (Bab 2)
- Kalkulasi Tagihan **Tahan Gangguan Parsial / ACID** (Bab 3)
- Integrasi perangkat keras **MikroTik RouterOS & FreeRADIUS** (Bab 4)
- Penjadwalan cerdas **Cron Node Sync** (Bab 5)
- Ilusi **SPA pada Mobile View (PWA)** yang mulus (Bab 6)
- Tembok **Keamanan Berlapis (AES, CSRF, Tokenisasi)** (Bab 7)

Keutuhan dokumen ini menjamin pemahaman sistematis bagi para pengembang (*Developer*) maupun Teknisi Jaringan ISP yang hendak melangsungkan (*maintenance*), membina (*scaling*), maupun memugar (refactor) modul di kemudian hari.
