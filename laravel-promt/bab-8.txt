## 8.1 Panduan Penyetelan Peladen Tingkat Lanjut (Advanced Server Tuning)

Aplikasi LadaPala-Bill memiliki beban kerja I/O (Input/Output) basis data yang cukup tinggi, terutama karena sistem pembacaan RADIUS yang masif (tabel `radacct`) serta *cron jobs* sinkronisasi yang beroperasi di latar belakang. Konfigurasi bawaan XAMPP atau peladen Linux murni (LAMP/LEMP stack) seringkali tidak memadai untuk menangani volume transaksi dari ribuan pelanggan secara serentak.

### 8.1.1 Penyesuaian MySQL / MariaDB (my.cnf / my.ini)
Tabel yang sering menerima operasi tulis (Write-Heavy) dalam aplikasi ini meliputi `billing_system_logs`, `local_site_sync_log`, `billing_isolir_queue`, dan terutama `radacct`. Untuk memastikan basis data (Database) tidak menjadi *bottleneck* (leher botol), Anda harus menyetel parameter InnoDB secara spesifik:

1.  **`innodb_buffer_pool_size`**: Secara krusial harus diubah menjadi 60% - 70% dari total RAM fisik peladen. Jika Anda mendedikasikan mesin virtual (VM) dengan 16 GB RAM, pastikan `innodb_buffer_pool_size=10G`. Ini memastikan seluruh *index* dari tabel `billing_customers` (yang sangat lebar) dapat disimpan secara persisten di memori.
2.  **`innodb_log_file_size`**: Diubah ke `512M` atau `1G`. Semakin besar log file InnoDB, semakin sedikit MySQL melakukan *flushing* I/O ke penyimpanan, sehingga *commit* atas transaksi pembayaran pada `billing_payment_save.php` (yang membungkus banyak baris `INSERT` pada tabel tagihan parsial) bisa berjalan instan.
3.  **`innodb_flush_log_at_trx_commit`**: Nilai bawaan adalah `1` (sangat aman). Untuk ISP skala menengah hingga besar, Anda dapat menyetel nilai ini ke `2`. Dengan nilai 2, *commit* akan menulis log ke *cache* sistem operasi, dan hanya menyimpannya (flush) ke perangkat penyimpan (*disk*) setiap satu detik. Hal ini meningkatkan performa insersi ganda hingga puluhan kali lipat dengan mengorbankan 1 detik resiko kehilangan log jika peladen mati mendadak (kernel panic/power loss).
4.  **`max_connections`**: Mengingat sifat LadaPala-Bill yang sering dikunjungi oleh skrip *Cron* berlapis secara serentak (Push, Pull, Queue, API), menaikkan *max_connections* ke `1000` (dari default 151) adalah suatu keharusan.

### 8.1.2 Penyesuaian PHP (php.ini)
Kerangka aplikasi dirancang untuk meminimalkan ketergantungan memori tinggi per sesi, namun tetap membutuhkan penyesuaian khusus:
*   `memory_limit = 512M`: Penting untuk pembuatan laporan bulanan dan pemrosesan `json_encode` berkapasitas raksasa dari data _Google Sheets Sync_.
*   `max_execution_time = 300`: Beberapa *script cron* seperti `cron_site_push.php` mungkin memakan waktu lebih dari standar 30 detik apabila sinkronisasi data (Upserting ratusan ribu baris log RADIUS) perlu melakukan *Catch-up* karena *site* lokal terputus berhari-hari.
*   `opcache.enable = 1`: OPcache PHP wajib dinyalakan di level produksi (Production). *Front controller* `index-shell.php` dan pustaka `RouterosAPI` berukuran sangat besar. Menyimpan _bytecode_ yang sudah dikompilasi ke memori (Memory Cached Bytecode) akan menekan CPU usase (penggunaan CPU) PHP-FPM secara substansial.

### 8.1.3 Penyetelan FreeRADIUS (radiusd.conf & mods-enabled/sql)
Modul ekstensi RADIUS dari sistem ini menuntut sinkronisasi yang presisi:
*   Buka konfigurasi modul SQL (biasanya terletak di `/etc/freeradius/3.0/mods-enabled/sql`).
*   Aktifkan fitur sambungan _Pooling_ (`pool { start = 5; min = 5; max = 50; }`) agar *daemon* RADIUS tidak merobohkan MariaDB dengan siklus inisialisasi koneksi setiap kali pelanggan _dial-in_ atau _dial-out_.

## 8.2 Struktur Antarmuka Pemrograman Aplikasi (API Payload Reference)

Dokumentasi ini menyoroti spesifikasi teknis dari dua layanan API yang berdiam di sistem: Endpoint Pembayaran dan Endpoint Konfigurasi Latar Belakang. Pengembang eksternal atau vendor (*3rd Party B2B Integrators*) sering kali membutuhkan referensi pasti terhadap skema JSON aplikasi ini.

### 8.2.1 REST API Endpoint: Pembayaran Tagihan (`POST /billing/billing_payment_save.php`)
Sistem ini menggunakan metode pembungkus HTTP POST tunggal dengan *payload* berformat *JSON* murni dan mewajibkan autentikasi *cookie-based*.

*   **URL**: `https://<domain>/billing/billing_payment_save.php`
*   **Header Wajib**: 
    - `Content-Type: application/json`
    - `Accept: application/json`
    - `Cookie: PHPSESSID=...`
*   **Body Permintaan (Request Payload Format)**:
```json
{
    "csrf_token": "a1b2c3d4e5f6... (token 64 karakter dari form sesi)",
    "customer_id": 1056,
    "bulan_bayar": ["2026-05", "2026-06"],
    "tgl_bayar": "2026-05-29 14:35:00",
    "metode": "transfer",
    "keterangan": "Lunas via Rekening BCA An. John",
    "diskon": 5000,
    "paid_amount": 295000
}
```

*   **Penjelasan Properti (Property Breakdown)**:
    - `csrf_token` (String, Mutlak): Diambil dari variabel `$csrf_token` yang di-_echo_ oleh `inc/csrf.php`. Kegagalan menyertakan nilai ini akan ditolak (Bounced) dengan HTTP Status `403`.
    - `customer_id` (Integer, Mutlak): Identitas relasional utama, yang mengacu pada tabel `billing_customers`.
    - `bulan_bayar` (Array of Strings, Mutlak): Memungkinkan pelunasan berganda (multi-month bulk payment). Format bulan harus `YYYY-MM`. Sistem _backend_ memvalidasi via Regex `/^\d{4}-\d{2}$/`.
    - `tgl_bayar` (String, Opsional): Memiliki dua skema _fallback_. Bila dikirimkan tanpa nilai jam (hh:mm:ss), sistem secara mandiri membubuhkan `$tglBayar = $tglBayar . ' ' . date('H:i:s')` agar data stempel waktu di dalam database MySQL tetap absolut (Real-time).
    - `paid_amount` (Float, Kondisional): Diperuntukkan bila pelanggan melakukan cicilan (Partial Payment). Jika `paid_amount` nilainya di bawah total tagihan (`baseAmount`), tagihan menjadi separuh lunas, dan ditandai oleh nilai logis `is_partial = 1` ke pangkalan data.

*   **Respons Normal (HTTP 200 OK - Response Structure)**:
```json
{
    "status": "success",
    "message": "Pembayaran 2 bulan (May 2026, June 2026) (diskon Rp 5.000) → Rp 295.000 untuk \"John Doe\" berhasil dicatat. Pelanggan terisolir telah diaktifkan otomatis karena tunggakan lunas.",
    "id": 9942,
    "receipt_group": "RG-20260529143500-a1b2c3",
    "payment_ids": [9942, 9943],
    "jumlah": 295000,
    "diskon": 5000,
    "subtotal": 295000,
    "ppn_amount": 0,
    "is_partial": false,
    "paid_amount": 295000,
    "tambahan_layanan": 0,
    "remaining_after": 0
}
```
*   **Analisis Respons**: Anda akan menyadari keberadaan properti `receipt_group`. Properti unik bertipe *String* (contoh: `RG-20260529143500-a1b2c3`) diinjeksikan secara transaksional pada seluruh elemen di tabel `billing_payments`. Properti ini merupakan benang merah bagi cetak faktur termal (Thermal Invoice Printer); sekalipun pelanggan membayar 5 bulan, kuitansi tunggal cukup mengikat dirinya pada nomor grup tanda terima (Receipt Group) tersebut tanpa harus mencetak lima buah nota kertas secara terpisah.

### 8.2.2 REST API Endpoint: Google Sheets Laporan Eksekutif (`GET /api/billing_finance_sync.php`)
Sistem eksternal, seperti _Google Apps Script_, menarik *(Pulling)* seluruh rekam jejak akunting teragregasi dari aplikasi LadaPala-Bill secara periodik. Layanan ini harus beroperasi pada HTTP _GET_ karena _Apps Script Server_ secara teknis memicu (triggering) *Cronjob* berbasis fungsi *UrlFetchApp*.

*   **URL**: `https://<domain>/api/billing_finance_sync.php?bulan=2026-05&tipe=all`
*   **Header Wajib**: 
    - `X-API-Key: BILLING_SYNC_KEY_20260529`
*   **Keamanan Eksternal (API Authentication Mechanism)**: 
    Tidak menggunakan sistem `cookie` atau otorisasi Berbasis Token (_Bearer Token_) tipe standar oAuth2. Sebaliknya, aplikasi membangkitkan kombinasi prase (Phrase) deterministik dari kalender tanggal berjalan (`BILLING_SYNC_KEY_{YYYYMMDD}`). Sistem pengaman ini secara brilian menghalau upaya manipulasi _Man-in-The-Middle_, dikarenakan kunci pengaman akan terus berevolusi (Rolling Key) dan hangus sesaat setelah pergantian fajar di esok hari.

    Format data yang diturunkan (Dumped) begitu masif, berakar pada struktur bersarang (*Nested Data*).

## 8.3 Pemecahan Masalah Ekstensif (Comprehensive Troubleshooting)

Modul-modul aplikasi memiliki kompleksitas berlapis karena mengintegrasikan komponen sistem operasi (Linux/Windows), peranti keras perute (MikroTik), dan layanan Cloud sinkronisasi DB (Hybrid-MySQL). Berikut adalah ensiklopedia malfungsi operasional yang jamak terjadi beserta prosedur penyembuhannya (Healing Procedures).

### Kasus 1: Pelanggan Sudah Melunasi Tagihan, Namun Koneksi Internet (PPPoE) Masih Berstatus Terisolir
**Deskripsi Masalah (Symptom)**: 
Seorang *Sales* (Admin Penagih) telah membubuhkan tanda "Lunas" (Mencetak Nota dan menekan Konfirmasi Pembayaran). Data di laporan (Laporan Keuangan) telah diperbaharui dengan baik. Namun di rumah pengguna (User), indikator sinyal merah (LOS) atau navigasi web pengguna tetap dialihkan ke Halaman Isolir (Isolir Landing Page).

**Analisis Alur Cacat (Traceback Route)**: 
Saat tombol "Lunas" ditekan, fungsi `billing_payment_save.php` melakukan pembaharuan pada basis data. Setelah tahap komit (`commit`), ia mencoba memeriksa nilai `isolated = 1`. Jika lunas, fungsi tersebut tidak mengeksekusi langsung *RouterOS API* secara *Synchronous* karena hal itu beresiko memblokir alur *Frontend* bila jaringan sedang terganggu (Lagging). Sebaliknya, ia menjejalkan data ke `billing_isolir_queue`. Masalah muncul bila _Cron_ Latar Belakang (Background Cron) mogok.

**Resolusi dan Tindakan Perbaikan (Corrective Action)**:
1.  **Cek Log Sistem (Audit Log Check)**: Buka modul Audit Log. Carilah galat "Gagal Konek ke Router (Timeout)". Jika baris log memaparkan pesan ini berulang kali, ada kemungkinan kabel fisik *Trunk* yang mengular ke Gardu/POP *MikroTik* sedang bermasalah, atau kredensial API MikroTik (Password Server) kadaluwarsa (berubah secara asimetris).
2.  **Uji Pemanggilan Manual (Dry-Run Queueing)**: Gunakan Command-Line (CLI) dan masuk ke direktori `/c/xampp/htdocs/bill/cron/`. Jalankan eksekusi paksa (Forced Execution):
    `php cron_isolir_queue.php`
    Apabila proses pemantauan log via Terminal mengabarkan "*Retry limit reached*", lakukan pemeriksaan ulang kata sandi `username/password` Router di halaman *Data Master > Router*.
3.  **Tindakan Penyelamatan Paksa (Force Fallback Mode)**: Jika Router MikroTik memang mati daya, dan teknisi sudah menghidupkannya, jalankan ulang *Cron* di atas. Antrean yang macet di tabel `billing_isolir_queue` akan kembali diproses dalam urutan kronologis yang sangat akurat.

### Kasus 2: Sinkronisasi Multi-Node (Cloud vs Local) Mengalami Split-Brain (Terbelah Data)
**Deskripsi Masalah (Symptom)**: 
*Sales* memasukkan pelanggan baru dari sistem ponsel aplikatif (Cloud Mobile View). Namun teknisi di kantor cabang (Local LAN-XAMPP) melaporkan bahwasanya mereka tidak bisa melihat pelanggan yang barusan ditambahkan, meski 3 jam telah berlalu.

**Analisis Alur Cacat (Traceback Route)**: 
Kegagalan bertitik kumpul (Converge) di atas diprakarsai oleh *File Mutex Lock*. Berkas pengunci `cron_site_push.lock` yang ditinggalkan oleh skrip sebelumnya tidak pernah dihancurkan (*released*). Ini biasanya terjadi ketika skrip `cron_site_push.php` mendadak diterminasi oleh Manajer Tugas Sistem Operasi (OS Task Manager OOM Killer) akibat RAM peladen penuh di tengah proses pertukaran (Transmitting) data ratusan megabita, meninggalkan fail `cron_site_push.lock` di direktori temporer OS (`sys_get_temp_dir()`).

**Resolusi dan Tindakan Perbaikan (Corrective Action)**:
1.  Buka direktori `/tmp` (atau `C:\Windows\Temp` pada Windows) pada Peladen Pusat (Cloud Server).
2.  Cari, dan hapus berkas kusam bernama `cron_site_push.lock`.
3.  Pastikan juga melihat log dari _Local Site Sync_. Cek tabel `local_site_sync_log`. Apakah ada deret catatan bertanda `status = 'error'` atau `status = 'partial'`. 
4.  Pemicu alternatif lainnya (Alternate Trigger) adalah inkonsistensi struktur (Schema Mismatch). Jika Anda baru saja meng-unggah (Upload) penambahan versi *Core Backend* baru ke *Cloud*, tapi lalai menjalankan berkas modifikasi pangkalan data (SQL Migration Script `update_schema_cloud.php`), berkas Sinkronisasi (*Cron Upsert Logic*) akan mendeteksi perbedaan kolom (Column Dissonance) lalu serta merta mengugurkan proses sinkronisasi untuk meminimalisasi kerusakan lintas peladen.

### Kasus 3: Peringatan Berbahaya "GOD_MODE Enabled - Masa Aktif Berakhir"
**Deskripsi Masalah (Symptom)**: 
Beberapa fungsi sistem tidak bekerja normal, dan pada _backend_ log tampak seruan `GOD_MODE=true` secara konstan kendatipun *Administrator* tidak pernah mengaktifkannya pada berkas konfigurasi.

**Analisis Alur Cacat (Traceback Route)**: 
Fungsi konfigurasi `config.php` telah dirancang secara *hard-code* (Terukir Kuat) dengan algoritme bom-waktu (*Time-bomb License Algorithm*). Perhatikan kode berikut:
`define('GOD_MODE', config_is_expired() ? true : config_bool('GOD_MODE', false));`
Fungsi `config_is_expired()` melakukan ekstraksi tanggal (Parsing Date) terhadap konstanta lingkungan (Environment Variable) `MASA_AKTIF`. Jika `strtotime('today') > $expiredTs`, sistem secara sepihak memaksakan aktivasi mode perlindungan (Kill-Switch).

**Resolusi dan Tindakan Perbaikan (Corrective Action)**:
1.  **Pembaruan Berkas Lingkungan**: Sunting (Edit) berkas rahasia `.env` di jalur pangkal (Root Directory). Temukan konstanta *Key-Value*:
    `MASA_AKTIF=2024-12-31`
2.  Ubah nilainya ke tenggat waktu masa depan, contohnya `MASA_AKTIF=2030-01-01`.
3.  Bila Anda memiliki akses modifikasi permanen terhadap sumber (Source Code Level Access), Anda dapat menonaktifkannya seutuhnya dengan membiarkan *value* (Nilai) pada variabel tersebut sepenuhnya kosong (Empty String), yaitu `MASA_AKTIF=`. Fungsi komparasi akan mengembalikan statuta (Status) *False* karena deteksi _Integer_ UNIX Timestamp gagal mencerna teks hampa.

### Kasus 4: Firebase Notifications (Push Notification App Mobile) Tidak Menyalak / Bisu
**Deskripsi Masalah (Symptom)**: 
Pelanggan mentransfer sejumlah pembayaran uang elektronik. Kasir memvalidasi, menekan OK. Berkas JSON Response di jendela peramban (Web Browser) mencetak status `success`, namun ponsel milik Pemilik Usaha ISP (*Owner*) yang meng-instalasi APK (Android Package) LadaPala-Bill tidak bergetar dan tidak membunyikan bel (Ting!).

**Analisis Alur Cacat (Traceback Route)**: 
Aplikasi LadaPala-Bill menyeberangkan (Bridge) _Event Observer_ notifikasi lewat perantara kelas Statis (Static Class) `FCMHelper::notifyAuditAdmins()`. 
Ada dua titik rapuh (Fragile Nodes) dalam orkestrasi notifikasi awan (Cloud Notification):
*   Tabel penyimpan keping token asimetris (`app_devices`) kotor. Keping *Hardware-ID* (FCM Token) milik gawai pintar (*Smartphone*) Sang Bos telah usang (Expired/Rotated by Google Play Services), tetapi gawai tersebut gagal melapor-ulang (Re-register) _Token_ terkininya melalui *Jembatan Javascript Web-View* ke Endpoint `register_device_token.php`.
*   Skrip kredensial _Service Account_ `inc/fcm_helper.php` kehilangan izin (Permission Denied) oleh *Google Firebase Server* karena *Server Key* lama, atau kredensial berkas berwujud JSON (Firebase Admin SDK) tidak tersedia.

**Resolusi dan Tindakan Perbaikan (Corrective Action)**:
1.  Instruksikan Sang Bos (atau *Sales* bersangkutan) untuk melakukan pemuatan ulang (Log out) aplikasi *Mobile*, tutup aplikasi (Swipe Close) dari *Recent Apps*, lalu buka dan otorisasi _Login_ ulang. Ini memicu ulang rutin algoritme *Android Native FirebaseService* untuk mengirim token mutakhir ke JS _Bridge_.
2.  Buka tabel MySQL `app_devices` dan pastikan baris atas nama (User ID) sang Bos telah memuat untai teks _Base64_ raksasa di dalam kolom `fcm_token`.
3.  Validasi berkas konfigurasi kunci asimetrik Firebase (`.env` -> `FCM_SERVER_KEY`). Jika Anda mengintegrasikan Firebase versi V1 (yang baru, yang mewajibkan _OAuth2 Bearer Token_), Anda wajib memperbarui pustaka pembungkus (_Wrapper Library_) `fcm_helper.php` agar selaras dengan arsitektur JWT (JSON Web Token) Google API v1.

## 8.4 Kesimpulan Arsitektur Rancang Bangun Peranti Lunak (Software Architecture Conclusion)

Menginjak akhir dari bedah teknis ekstensif (Extensive Technical Walkthrough) ini, kita bisa merekapitulasi (Recapitulate) arsitektur monumental yang dikandung dalam sistem SISPEMB (LadaPala-Bill).

*   **Pola Desain Hibrida (Hybrid Design Pattern)**: Aplikasi ini tidak tunduk secara utuh pada dogma desain berbasis pola (Pattern-based Design) kaku seperti *MVC (Model-View-Controller)* konvensional (layaknya Laravel atau CodeIgniter). Sistem lebih menganut paradigma *Front-Controller Procedural (Micro-services via File Routing)*. Eksekusi program diinisiasi dari `index-shell.php` dan dijembatani via `app-gateway.php`, membentuk struktur sistem yang tangkas (*Agile*) dan tak rentan (*Resilient*) terhadap kebocoran memori dari perpustakaan raksasa yang tak terpakai (Unused Monolithic Library).
*   **Pemodelan Data Non-Destruktif (Non-Destructive Data Modeling)**: Sistem menghindari praktik *Hard Delete* (Hapus Mutlak). Entitas yang usang tidak dimusnahkan. Ketika terjadi pembaharuan tagihan yang pelik, penggunaan pendekatan *Soft-State Update* dan penguncian baris (`FOR UPDATE`) di InnoDB memanifestasikan komitmen kuat terhadap prinsip-prinsip *Transaksional ACID* tingkat perbankan, menjadikan kehilangan aset moneter akibat kolisi transaksional (*Race Conditions*) hampir mustahil secara teknis.
*   **Infrastruktur Mandiri (Self-Sustaining Infrastructure)**: Pada ujung kapabilitas otomasi, ketergantungan minimum terhadap perangkat lunak perantara (Middleware) mencolok. Semua pekerjaan latar belakang, pelokasian antrean (*Queue Polling*), hingga interkoneksi dupleks (Bidirectional Sync) dilakukan di dalam perut pangkalan data (MySQL Base) itu sendiri, ditemani dengan panggilan serentak *CLI PHP Cron*.

Sistem LadaPala-Bill ini bukan sekadar portal administratif semata, melainkan instrumen *Enterprise Resource Planning (ERP)* mini, yang dirajut khusus (Tailor-Made) demi memastikan denyut nadi bisnis para pelakon Usaha Jaringan Internet Mikro (RTRW Net & ISP Menengah) tak henti berdenyut, mengantarkan layanan tagihan secepat kedipan *Fiber Optic*, dan senyaman pesan selamat pagi yang singgah di layar gawai pintar.
